Secret Shuttle Demo
Scene 00 / 09 Magic path
magic path

One verb. One approval. Done.

provision --infer reads .env.example + your framework configs (Vercel, Supabase, GitHub Actions), writes a plan, and mints one batch approval. You click Approve once in the hub. provision --continue ships every secret to every destination — and audit hands you the receipt.

Watch for → The agent never sees a value. One human click covers the whole batch. The session checkbox lets the next matching push skip the popup for 15 minutes.
The 30-second loop
This is the day-to-day path. The nine scenes that follow are the advanced low-level mechanics — browser marks, blind-mode reveal-capture, inject-submit — the escape hatch for non-templated providers. Most provisioning never leaves this scene.
stuck

"…I need the secret. Can you paste it?"

The dev finished a Stripe integration in their agent. Time to ship. They ask Claude Code to deploy to prod — and immediately hit the secret wall.

Pasting a webhook signing secret into a chat is how secrets land in scrollback, git history, and ~/.zsh_history. Nobody wants that, but the path of least resistance is exactly that.

Watch for → Claude refuses to handle the secret itself, then bounces the work back to the human. No good options.
Gap to spot
Does the dev know why pasting is bad? A "why this matters" one-liner could land Secret Shuttle in 5 seconds vs 5 minutes of search.
curious

A bridge — refs, not values.

The dev points Claude at the Secret Shuttle skill URL. Claude reads it, summarises the model: a local daemon holds the vault, the agent works through refs like ss://stripe/prod/…, and only the human sees raw values during approval.

Watch for → The dev tells the agent. The agent doesn't proactively suggest it (yet).
Gap to spot
How does the dev actually hear about Secret Shuttle? Skill registries? Cursor / Codex marketplaces? A "Claude recommended it" path is the dream — but only if every agent ships with awareness of it.
setting up

One command, then passphrase.

Claude runs npx secret-shuttle init — one command that installs the daemon, starts it, and opens the passphrase window. The CLI never reads the passphrase. A local web window opens — the human types it directly into the daemon's UI.

Watch for → First-time unlock = "create passphrase" mode. Same window, different verb.
Gap to spot
Is the difference between "create" and "enter" obvious enough in the unlock window? Today the only signal is one line of copy.
wiring agent

Teach the agent the protocol.

init already wrote the Secret Shuttle skill for detected agents. From here, the agent knows the order — status first, prefer template run, never log raw values, surface manual_recovery_required to the human.

Watch for → Add more agents anytime: secret-shuttle agent install codex — also cursor, copilot — marker-managed so it doesn't clobber the rest of the file.
Gap to spot
Does Claude actually re-read its skills mid-session, or does it need a new session to pick the file up? The demo should be honest if it's the latter.
first win

Generate & push — two clicks.

The dev asks for an INTERNAL_CRON_SECRET in prod, on Vercel. Claude runs status, secrets set, then template run vercel-env-add. Two approval cards in sequence — generate, then publish.

Watch for → Each approval is scoped to a single action. No grant is wider than what the human just OK'd.
Gap to spot
Two approval clicks for one user intent — is that friction worth it for the audit trail, or could there be a single "this whole intent" composite approval?
marking targets

Browser opens. Marks get placed.

For the Stripe webhook handoff, Claude runs browser start. A daemon-owned Chrome window opens. Claude navigates to the webhook settings, then marks the reveal button and the secret field by opaque label.

Watch for → The daemon stores only structural metadata for each mark — no selectors, no DOM text, nothing the agent can fingerprint.
Gap to spot
The mark pick choreography (run command, then click in the browser) is delicate. Worth a per-agent runbook so agents don't fumble the dance.
blind mode 🔒

Reveal. Capture. Verify gone.

Claude calls reveal-capture. The daemon takes the wheel — blind mode engages, the daemon clicks reveal, captures the bytes from the marked scope, hides them, and proves the value is absent from every surface the daemon can observe in the DOM before resuming.

Watch for → The approval card shows the reveal label, capture mode, hide handle, and an explicit auto-resume disclosure (red).
Gap to spot
How does the dev visually know blind mode is on? Today it's CLI status. A daemon-painted overlay in Chrome itself would be reassuring.
handoff

Inject. Submit. Hand back.

Same browser, now on Vercel. Claude marks the value field + Save button, then calls inject-submit with a success marker. The daemon writes the secret, clicks save, waits for the marker, proves the secret is gone from the DOM surfaces it can observe, and only then hands observation back.

Watch for → The agent never sees the value. The argv never carries it. The success text is what the human approved.
Gap to spot
Marked field + marked submit + success text is a lot of ceremony. For the four templated providers, template run is much less work — should it be the default the skill steers agents toward?
done

All green. No one saw it.

Status reports a clean state. The Stripe webhook signing secret is in Vercel production env. The dev never saw the raw value. The agent never saw it. No argv, no env, no log line, no chat transcript ever carried the bytes.

Watch for → The closing summary references the secret by ss:// ref and fingerprint — that's the agent's vocabulary from now on.
Gap to spot
Where does the dev go next? Rotation? More providers? A "what now" CTA in the CLI itself would close the loop — right now the demo ends, the product is silent.
