## §0 — Cross-section context

### What this burst is for

Two security-infrastructure plans the earlier bursts forward-referenced:

- **Plan 5s — per-project agent IDs (opt-in):** today every project on a machine shares one `<runtime>-<machineId>` identity. Owner-enforced consumption (Burst 4) therefore cannot distinguish "agent in project A" from "agent in project B" — a session-affordance leak in one project is consumable by the same agent id in another. 5s adds an opt-in per-project dimension to the derived identity.
- **Plan 5q — plaintext-out-of-heap memory hygiene:** secret values currently live as JS `string`s (`SecretRecord.value: string`). JS strings are immutable and cannot be zeroed — they linger in the daemon heap until GC, so a core dump / heap snapshot of the daemon process can contain recently-handled secret values. 5q moves the per-secret *use path* to a scrubbable `Buffer`, wrapped in a redaction-safe `SecretValue` type.

**Plan 5a (native keychain) is NOT in this burst — it already shipped.** During scoping, the keychain adapters (`src/vault/keychain/{darwin,linux,windows}.ts`) were found to be complete `@napi-rs/keyring`-backed implementations (not stubs), with the enroll/disable routes, the `keychain_opt_out` envelope flag, opportunistic enrollment, and the unlock-first keychain read all wired (labeled "Plan 5b/5f" in `unlock-session.ts`). `@napi-rs/keyring@^1.3.0` is in `package.json` and installed; keychain tests pass. The only 5a residue is **stale doc comments** in `src/vault/keychain/index.ts` and `types.ts` that still describe the implementations as "stubs in Plan 1 / Plan 5a wires in the real implementations." Correcting those comments is folded into this burst's wrap (Wrap.1) — pure documentation lint, no functional change.

### What changes vs. what does not

**Unchanged (hard constraints):**
- The daemon trust boundary: the agent never sees plaintext **except where the agent itself explicitly asked for the rendered output** — i.e. `inject-render -o -` (stdout-passthrough mode), which renders a plaintext template string and returns it in the response body `content` field after all resolve/policy/approval checks pass (`inject-render.ts:99`, `inject-render.ts:108`; it flips `valueVisibleToAgent = true` first). That is the **one** existing, intentional plaintext-out wire surface (an agent-requested render, not a leak), and file mode (`-o <abs-path>`) writes to a 0600 file instead. 5q is a heap-lifetime hardening, NOT a leak fix — verification confirmed no daemon JSON *response* serializes a secret value today **other than that explicit `inject-render -o -` passthrough**; every other `.value` read flows to a daemon-internal sink (Chrome injection, child stdin, 0600 env-file, HMAC fingerprint, absence-proof).
- **Wire formats:** 5q is an internal in-memory representation change. No HTTP request/response shape changes. No `error_code` registry changes beyond any additions §1/§2 explicitly introduce (none expected — see each section).
- **On-disk vault format:** `VaultPlaintext` stays JSON-serialized then AES-256-GCM-encrypted (`encryptVault`/`decryptVault` unchanged). 5q is Tier A (per-secret use path); the JSON vault format is explicitly left alone (see §2 "Boundary").
- Approval semantics, session affordance, audit log format, the `ss://` ref grammar, every Burst 4–6 surface.
- **Existing fingerprints stay valid.** `fingerprintSecret` is an HMAC over the value's bytes; HMAC(utf8-bytes-of-string) === HMAC(Buffer-of-same-bytes). Changing the parameter type from `string` to bytes does NOT change any computed fingerprint, so no stored fingerprint migration is required.

**Changed:**
- **5s (opt-in, additive):** `deriveAutoAgentId` gains an optional third `projectScope` parameter (back-compatible — 2-arg callers get byte-identical output). Opt-in via `identity.perProject` in `secret-shuttle.config.json` and a `secret-shuttle init --per-project-identity` flag. No existing user's identity changes unless they opt in.
- **5q (internal hardening):** new `SecretValue` wrapper class; a new plaintext-resolving accessor (`resolveSecret`/`resolveRefs`) returns a disposable `ResolvedSecret` whose `value` is a `SecretValue`; a **no-value metadata accessor** (the existing `Vault.inspect`→`AgentSecretMetadata`, or a thin `getSecretMetadata` wrapper) serves metadata/existence callers so they hold **no plaintext at all**; the string-valued stored `SecretRecord` is reserved for **vault internals only** (crypto serialize/deserialize, rotation/fingerprint-recompute). The per-secret consumers read `.bytes()` and scrub after use; `fingerprintSecret`/`fingerprintMatches` take bytes. The on-disk `StoredSecretRecord.value` stays `string`.

### Sequencing

5s first (low risk, localized), then 5q (cross-cutting, ~55 `.value` call sites). Wrap last (5a doc fix + CHANGELOG + version bump). One codex gate per stage (spec, plan, impl).

## §1 — Plan 5s: per-project agent IDs (opt-in)

### Current state (verified)

`src/daemon/auth/agent-id.ts`:
```ts
export function deriveAutoAgentId(runtime: string, machineId: string): string {
  const digest = createHash("sha256").update(`${machineId}\x00${runtime}`).digest("hex");
  return `${runtime}-${digest.slice(0, 16)}`;   // e.g. "claude-7f2a9c1b3d4e5f60"
}
```
Called once, at `src/cli/commands/init.ts:238`, inside the per-runtime token-mint loop. `machineId` is read from `<SHUTTLE_HOME>/machine-id`. The derived id must satisfy `AGENT_ID_RE = /^[a-z][a-z0-9._-]{0,63}$/` (asserted by `assertAgentIdValid`). `"root"` and `"daemon"` are reserved.

### Change

**Back-compatible signature extension:**
```ts
export function deriveAutoAgentId(runtime: string, machineId: string, projectScope?: string): string {
  const material = projectScope === undefined
    ? `${machineId}\x00${runtime}`                       // unchanged — 2-arg callers get identical output
    : `${machineId}\x00${runtime}\x00${projectScope}`;   // per-project variant
  const digest = createHash("sha256").update(material).digest("hex");
  return `${runtime}-${digest.slice(0, 16)}`;
}
```
The id format is unchanged (`${runtime}-${16 hex}`); only the digest material gains a scope component when opt-in is on. Length and `AGENT_ID_RE` validity are preserved.

**`projectScope` resolution** — new helper in `agent-id.ts`:
```ts
import { execFileSync } from "node:child_process";

/** Absolute git-repo-root path, or process.cwd() when not in a repo.
 *  Hashed into the agent id — the path itself never appears in the id. */
export function resolveProjectScope(cwd: string): string {
  try {
    const root = execFileSync("git", ["rev-parse", "--show-toplevel"], {
      cwd, encoding: "utf8", stdio: ["ignore", "pipe", "ignore"],
    }).trim();
    return root.length > 0 ? root : cwd;
  } catch {
    return cwd;   // not a git repo, or git absent → cwd is the scope
  }
}
```

**Opt-in mechanism (two equivalent triggers, config is canonical):**
- **Config:** `identity: { perProject: true }` in the project's `secret-shuttle.config.json` (the same file Burst 6 introduced `infer.supabaseNames` into). Read via a small loader (mirror Burst 6's `loadInferConfig` defensive pattern: missing file / malformed JSON / non-object / non-boolean `perProject` → treat as `false`).
- **Flag:** `secret-shuttle init --per-project-identity` — when passed, `init` behaves as if `identity.perProject` were true for this run AND writes `identity: { perProject: true }` into `secret-shuttle.config.json` so subsequent runs are consistent (create the config file if absent; merge the key if present, preserving `infer.*`).

**Where it lands:** `init.ts` (the one caller at line 238) reads the opt-in (config OR flag); when true it resolves `projectScope = resolveProjectScope(process.cwd())` and calls the 3-arg `deriveAutoAgentId`. When false it calls the 2-arg form (today's behavior). The token-mint + settings-write path is otherwise unchanged — it writes whatever id `init` derived.

### Known-unknown resolutions

- **Monorepo:** sub-projects under one git root share an agent id (one repo = one trust domain). Documented in the flag help + CHANGELOG. Finer (per-subdir) granularity is a future opt-in, not this burst.
- **Relocation:** moving the project directory changes the git-root path → changes the derived id → orphans that project's sessions/grants. With opt-in this is the user's informed choice; re-running `init` after a move re-derives and re-mints. Documented in the flag help + CHANGELOG.
- **Determinism:** same (machine, runtime, project path) always yields the same id (pure hash) — no state file needed beyond the existing machine-id.

### Error codes

None new. 5s is additive opt-in behavior over an existing derivation; failure modes (git absent, not a repo) fall back to cwd rather than erroring.

### Tests (TDD)

- `deriveAutoAgentId(runtime, machineId)` (2-arg) output is **byte-identical** to the pre-change function for a fixed input (regression pin — guarantees existing users unaffected).
- `deriveAutoAgentId(runtime, machineId, scope)` (3-arg) produces a distinct id from the 2-arg form, stable for a fixed scope, and `AGENT_ID_RE`-valid.
- Two different `projectScope` values → two different ids; same scope → same id.
- `resolveProjectScope`: returns the git-root in a temp git repo; returns cwd in a non-repo temp dir; returns cwd when `git` errors.
- Config loader: `perProject: true` honored; missing/malformed config / non-boolean → `false` (defensive).
- `init --per-project-identity` writes/merges `identity.perProject` into `secret-shuttle.config.json` without clobbering an existing `infer.*` block.

## §2 — Plan 5q: `SecretValue` + Buffer use-path (Tier A)

### Goal & boundary

Keep long-lived plaintext copies out of the daemon heap by making the per-secret **use path** a scrubbable `Buffer`, wrapped in a redaction-safe `SecretValue`. **Tier A:** the on-disk JSON vault format is untouched. The bulk `JSON.stringify(VaultPlaintext)`→encrypt / decrypt→`JSON.parse` path (`crypto.ts`) inherently materializes a plaintext string of all values: `decryptVault` builds the full plaintext via `Buffer.concat([...]).toString("utf8")` then `JSON.parse`s it into stored secret strings (`src/vault/crypto.ts`), and `encryptVault` runs `JSON.stringify(plaintext)`. Eliminating that requires a binary vault format + on-disk migration (Tier B), which is **explicitly out of scope** (see §3) and documented as an accepted, bounded limitation.

**Honest scope of the Tier A win (verified — do not overstate).** The vault read/write transient is *bounded* but NOT purely synchronous and NOT confined to `JSON.stringify`: `decryptVault` materializes every stored value as a JS string on read (decrypt → `toString("utf8")` → `JSON.parse`), and `upsertSecret` stores `record.value` as a string and `await`s `this.write(plaintext)` (`src/vault/vault.ts`) — so the materialized stored strings live across the awaited disk write before being dropped to GC. The win 5q actually captures is therefore: **it removes the *extra, long-lived* per-consumer plaintext string that today persists after the resolve boundary for the duration of async browser injection / child-process lifetime / absence-proof.** It does NOT remove (a) the vault-operation-scoped stored strings that `decryptVault`/`upsertSecret` materialize (those are Tier B), nor (b) the strings re-materialized *at* the accepted platform/protocol boundaries enumerated in §2's "Sink reality" — **browser/CDP** (inject out + capture in), **template render** (`inject-render` builds a `Map<string,string>` and `render` returns a plaintext string; in stdout mode that string is the agent-requested response `content`), and **child env** (`run-resolve` assigns secrets into a string-valued `NodeJS.ProcessEnv`). The other internal sinks — child stdin, tmp env-file, masker — go Buffer-native under 5q. Note these accepted-boundary strings are **not uniformly short-lived**: the browser/CDP and template-render transients are dropped right after the sink call, but the **child env strings live for the child's lifetime** (inherent to passing secrets via environment) and the **stdout-render `content`** leaves the daemon in the response body. Tier A shrinks plaintext heap lifetime from "as long as the consuming op runs" down to "the vault read/write op **plus** the explicitly-audited platform/protocol-boundary string windows" — i.e. no plaintext lingers between vault resolution and the sink call, and the only remaining strings are the bounded ones each enumerated and classified in §2.

### `SecretValue` — the guard-by-construction

New file `src/vault/secret-value.ts`:
```ts
import { timingSafeEqual } from "node:crypto";
import { inspect } from "node:util";

const REDACTED = "[secret]";

/**
 * A secret's plaintext bytes, wrapped so accidental stringification redacts
 * instead of leaking. The ONLY way to read the bytes is `.bytes()`, which is
 * greppable + auditable. toString/toJSON/inspect all return "[secret]", so
 * `${sv}`, JSON.stringify(sv), console.log(sv), and template/log interpolation
 * cannot leak the value. Call dispose() after use to zero the backing Buffer.
 */
export class SecretValue {
  #buf: Buffer;
  #disposed = false;

  private constructor(buf: Buffer) { this.#buf = buf; }

  static fromUtf8(s: string): SecretValue { return new SecretValue(Buffer.from(s, "utf8")); }
  static fromBuffer(b: Buffer): SecretValue { return new SecretValue(Buffer.from(b)); } // defensive copy

  /** The plaintext bytes. Throws if already disposed. The single audited door. */
  bytes(): Buffer {
    if (this.#disposed) throw new Error("SecretValue used after dispose()");
    return this.#buf;
  }

  /** Byte length (safe to expose — not the value). */
  get byteLength(): number { return this.#buf.length; }

  /** Constant-time compare against another secret's bytes. */
  equals(other: SecretValue): boolean {
    const a = this.bytes(), b = other.bytes();
    return a.length === b.length && timingSafeEqual(a, b);
  }

  /** Zero the backing buffer. Idempotent. */
  dispose(): void { this.#buf.fill(0); this.#disposed = true; }

  toString(): string { return REDACTED; }
  toJSON(): string { return REDACTED; }
  [inspect.custom](): string { return REDACTED; }
}
```

### Type split

- **On-disk / in `VaultPlaintext` — unchanged, vault-internal only:** `SecretRecord.value: string` (the *stored* record; the existing `SecretRecord` type keeps `value: string` so `encryptVault`/`decryptVault` and the upsert-write path serialize cleanly). The crypto layer and vault-internal rotation/fingerprint-recompute keep operating on the stored string. **This string-valued record is no longer handed to daemon routes** — only the two accessors below cross the vault boundary.
- **Returned to metadata/existence callers — no plaintext:** the existing `Vault.inspect(ref) → AgentSecretMetadata` (no `value` field; carries `ref`/`environment`/`allowed_domains`/`allowed_actions`/`fingerprint`), or a thin `getSecretMetadata` alias. Metadata callers hold neither a plaintext string nor a `SecretValue`.
- **Returned to plaintext consumers — new:** a *plaintext-resolving* accessor returns a `ResolvedSecret` — all the `SecretRecord` metadata fields EXCEPT `value` is a `SecretValue` (created via `SecretValue.fromUtf8(stored.value)` at the resolve boundary). Consumers never touch the stored string; they get the scrubbable, redaction-safe `SecretValue`.

```ts
export type ResolvedSecret = Omit<SecretRecord, "value"> & { value: SecretValue };
```

**Critical: `getSecret` is NOT a clean plaintext boundary today (verified), and its string `value` is dead weight for the callers that only read metadata.** Many `getSecret` callers never consume the plaintext — they read metadata/existence only — yet they currently receive (and hold in heap) a plaintext `value: string`. This is **especially bad for production delete/rotate**: both fetch the full record *before* `requireApprovals` (`secrets-delete.ts:50`, `secrets-rotate.ts:50`, reading only `environment`/`allowed_domains`), so `record.value`/`oldRecord.value` lives across human-approval latency for no reason. The metadata callers must hold **no plaintext at all**:

- **Metadata/existence-only callers (→ no-value metadata, no `SecretValue`, no string):** `secrets.ts:154` (generate overwrite-scope lookup → reads `allowed_actions`), `secrets-import.ts:103` (existing-check), `secrets-delete.ts:50` (existence guard + production branch), `secrets-rotate.ts:50` (reads `environment`/`allowed_domains`), and the compare route (reads `fingerprint` only — see Consumer-changes note).
- **True plaintext-use callers (get a `ResolvedSecret` with a disposable `SecretValue`):** `inject-submit.ts:53` (inject), `secrets.ts:394` (inject — `inject_into_field`; note `secrets.ts:484` is the **compare** route, which is metadata-only and listed above, NOT inject), `templates.ts:130` (child stdin), plus `resolveRefs`/`resolveWithinSecrets` consumers and `run-resolve` (env). **`reveal-capture` is NOT here — it does not resolve any stored secret (no `getSecret`/`resolveRefs` call); it is an inbound capture *producer* (`upsertSecret` at `reveal-capture.ts:424`), listed under the Inbound (write) path below.**

Therefore **split the accessor into three shapes** — never return a plaintext string to a non-vault caller:
- **No-value metadata accessor** for the metadata/existence paths: reuse the existing `Vault.inspect(ref) → AgentSecretMetadata` (which already carries `ref`/`environment`/`allowed_domains`/`allowed_actions`/`fingerprint` and **no `value`**, and already throws `secret_not_found` for missing *or* soft-deleted refs — identical existence semantics to `getSecret`), or add a thin `getSecretMetadata` alias. Migrate the metadata callers above off `getSecret`. They allocate neither a `SecretValue` nor a plaintext string.
- **Plaintext-resolving accessor** (e.g. `resolveSecret(ref)` / the existing `resolveRefs`) that returns an **owned, disposable** `ResolvedSecret`; only the true plaintext-use APIs call it, and each is responsible for `dispose()` after the sink resolves.
- **Vault-internal stored-record access** (the string-valued `SecretRecord`): reserved for vault internals only (crypto serialize/deserialize, rotation/fingerprint-recompute). The string `value` never leaves the vault layer — no daemon route holds a plaintext string after this split.

> Note on `assertSecretActionAllowed` callers: the metadata callers that also enforce policy (e.g. the policy check reads `ref`/`allowed_actions`) are covered by the policy-parameter widening below — `AgentSecretMetadata` and `ResolvedSecret` both satisfy `Pick<SecretRecord, "ref" | "allowed_actions">`, so no caller needs the stored string to type-check.

**Policy type widening (verified):** `assertSecretActionAllowed(secret: SecretRecord, action)` in `src/policy/policy.ts` requires a `SecretRecord` and is called on inject/rotate paths. Since those paths may now hold a `ResolvedSecret`, widen the policy parameter to the structural subset it actually reads — `Pick<SecretRecord, "ref" | "allowed_actions">` (both present on `ResolvedSecret` since `Omit` only drops `value`) — so the function type-checks against both `SecretRecord` and `ResolvedSecret` without forcing callers to carry the stored string.

### Consumer changes (the resolved-plaintext use sites)

Each reads `.value.bytes()` (instead of `.value`) and scrubs (`.value.dispose()`) after the value is no longer needed:
- `inject-submit.ts` (`browser.injectField`, `proveAbsence`)
- `inject-render.ts` (`valuesMap` → `parsed.render(...)`; see note below — this is an existing plaintext-out wire surface in stdout mode)
- `templates.ts` (child stdin)
- `secrets.ts` (`injectFocused` at `secrets.ts:394`)
- `run-resolve.ts` (env values → child `NodeJS.ProcessEnv`; see Sink-reality note — env values are an **accepted string boundary** (platform env API is string-only), the masker built from the same values is Buffer-native)

**Two-phase late-resolve discipline (REQUIRED for every true-plaintext consumer — not just `compare`/`import`).** Verified: today *all* of these routes resolve the plaintext **before** their approval gate and use it after — `inject-submit.ts` `getSecret` at `:53` → `secret.value` at `:171` (requireApprovals at `:124`); `/v1/secrets/inject` (`secrets.ts`) `getSecret` at `:394` → `injectFocused(secret.value)` at `:445` (requireApprovals at `:422`); `templates.ts` `getSecret` at `:130` → child-stdin use (requireApprovals at `:198`); `run-resolve.ts` `resolveRefs` at `:220` → `env[...] = record.value` at `:337` (requireApprovals at `:302`); `inject-render.ts` `resolveRefs` at `:54` → `render` at `:97` (requireApprovals at `:80`). A naive migration that merely swaps the **pre-approval** `getSecret`/`resolveRefs` call for a `resolveSecret`/`SecretValue`-returning `resolveRefs` at the **same** call site would hold a `SecretValue` (live plaintext bytes) across human-approval latency — directly violating the §5 invariant "no candidate/secret value is held across an approval gate." 5q therefore requires the same two-phase split already spelled out for `compare`/`import` to be applied to **each** of these consumers:
  1. **Before approval — metadata only.** The pre-approval policy/domain/binding preflight reads only non-secret fields (`assertSecretActionAllowed` needs `ref`/`allowed_actions`; `enforceDomain`/binding need `ref`/`environment`/`allowed_domains` + the already-available page domain), so the route fetches the **no-value metadata accessor** (`Vault.inspect`→`AgentSecretMetadata` / `getSecretMetadata`) before `requireApprovals` — **no `SecretValue` and no plaintext string is allocated yet.** (`resolveRefs` consumers — `run-resolve`, `inject-render` — likewise do the per-ref `assertSecretActionAllowed` + production-classification preflight off metadata, deferring the plaintext resolve.)
  2. **After approval, immediately before the sink — resolve plaintext.** Once `requireApprovals` resolves, the route calls the plaintext-resolving accessor (`resolveSecret(ref)` / the `SecretValue`-returning `resolveRefs`) to obtain the owned, disposable `ResolvedSecret`(s) **immediately before** the sink call (the CDP inject, the child-stdin write, the `env[...]=` assignment per the run-resolve drop-reference rule, or the `valuesMap.set` per the inject-render file-mode reorder), converts `.bytes()`→string only at that sink, and **disposes each `SecretValue` in a `finally`** so the bytes are scrubbed on the success, throw, and timeout paths alike. (For `inject-submit`/`/v1/secrets/inject` this means the `getSecret(ref).value` read moves to a `resolveSecret(ref)` call placed after the `requireApprovals` await and the blind-window/revalidation steps, right before `injectIntoBackendNode`/`injectFocused`; dispose in the route's outer `finally`.) **`inject-submit` is a deliberate *multi-sink* retention case (verified — do not dispose after the first sink).** Its `SecretValue` feeds **two** sinks: the inject (`injectIntoBackendNode(..., secret.value)`, `inject-submit.ts:171`) and, after a success-text wait that can run up to `successTimeoutMs` (default 15s, capped at 60s — `SUCCESS_TIMEOUT_DEFAULT_MS`/`SUCCESS_TIMEOUT_CAP_MS` at `inject-submit.ts:27-28`; `observeText` at `inject-submit.ts:215`), the **absence proof** `proveAbsence(secret.value)` (`inject-submit.ts:222`). The two-phase rule for this route therefore holds the single owned `SecretValue` across **both** sinks — i.e. a *bounded `SecretValue` retention window that spans the inject, the up-to-`successTimeoutMs` `observeText`, and the absence proof* — and disposes it **once** in the route's outer `finally` (after the proof, not after the inject). This is the spec's chosen strategy because `proveAbsence` must search the page for the **exact injected value** — disposing after the inject would leave nothing to prove absence of, and re-resolving for the proof would re-allocate the same plaintext bytes anyway. The retention is bytes-in-`SecretValue` (redaction-safe, scrubbed on the throw/timeout paths via the same outer `finally`), strictly shorter than today's plaintext-`string` lifetime which already spans the identical window. For the conditional gates (`run-resolve` and `inject-render` only call `requireApprovals` when a production ref is present — `run-resolve.ts:300`, `inject-render.ts:65`), "after approval" means **after the conditional gate point** in all paths: resolve the plaintext after the `isProduction`/`bindings.length>0` block (whether or not it awaited an approval), never before it — so the non-production path (no gate) still defers the `SecretValue` allocation to immediately before the sink, and the production path never holds it across the gate.
This makes every true-plaintext consumer hold plaintext only in the window between vault resolution and the sink — never across the approval gate — consistent with the `compare` late-capture reorder and §5 success-criterion (ii). The accepted-string-boundary classification (browser/CDP, template render, child env) is unchanged; this rule is about **when** the `SecretValue` is created and disposed, not about which sinks stay string-typed. The ordering tests in §2 "Tests" are extended to cover these consumers (see the lifetime-guarantee tests).

(**Not a resolved-consumer:** `reveal-capture.ts` *captures* a value and stores it via `upsertSecret` — it is an inbound *producer*, handled in the Inbound (write) path below, not a `.value.bytes()` read site.)

**`compare` is NOT a resolved-plaintext consumer.** It does not resolve or read any stored secret value — it reads the stored record's `fingerprint` (metadata) and HMACs the *browser-captured candidate* (`fingerprintMatches(capture.value, secret.fingerprint, fpKey)`, `secrets.ts:515`). So compare needs only the metadata accessor (no `SecretValue`, no dispose) plus a `Buffer.from(capture.value, "utf8")` for the captured candidate. See the fingerprint-signature note below.

**Required compare capture reorder (verified — current order holds the captured candidate across approval latency).** Today the compare route captures the page value (`capture = await browser.captureFocused()/captureSelection()`, `secrets.ts:486-488`) **before** awaiting `requireApprovals` (`secrets.ts:505-512`), then HMACs it at `secrets.ts:515` — so the browser-captured candidate **string** lives in the daemon heap across human-approval latency (arbitrarily long), contradicting "capture transient materialized only at the sink call and dropped right after." 5q therefore requires **gathering only the non-secret preflight data before approval** — the captured **domain** (for the `domain`/`allowed_domains`/`enforceDomain` checks at `secrets.ts:489-492`) and the `ApprovalBinding` (which needs only metadata + `capture.domain`, `secrets.ts:494-504`) — and **capturing the candidate value itself only after `requireApprovals` resolves**, immediately before the `fingerprintMatches` HMAC. Concretely: for the focused path a no-value pre-approval read exists at the *type* level — `BrowserOps.readFocusedFingerprintAndDomain(): Promise<Omit<CaptureResult, "value">>` (`internal-ops.ts:179`) returns domain/fingerprint without the value — **but it is NOT value-free at the heap level today (verified — must be fixed, not relied on as-is):** its implementation (`internal-ops.ts:957`) evaluates the same `READ_SCRIPT` as `captureFocused`, and `READ_SCRIPT` (`internal-ops.ts:209-214`) materializes `value` (`window.getSelection()?.toString()` / `a.value` / `a.innerText`) into the returned object. The TypeScript generic at `internal-ops.ts:959` omits `value`, but the CDP `Runtime.evaluate` result still surfaces the full object — including the plaintext `value` — in the daemon heap before the function declines to return it. So as a pre-approval preflight it does **not** actually keep the candidate plaintext out of heap across the approval gate. **5q therefore requires a genuinely metadata-only path for the pre-approval preflight** — a dedicated metadata-only in-page script (a `READ_META_SCRIPT` that returns `{ ok, field, domain, title, urlHost }` and never reads selection text / `.value` / `.innerText`), or an equivalent CDP read that returns no `value` field — so `readFocusedFingerprintAndDomain` materializes no candidate plaintext on the daemon heap; the candidate value is then pulled only after approval via `captureFocused()`/`captureSelection()` (the existing value-bearing `READ_SCRIPT` path). The selection path either gains an analogous no-value metadata read or re-issues `captureSelection()` after approval (one extra CDP round-trip) — so no candidate-value `Buffer`/string **and no in-heap candidate plaintext from the preflight** is held across the approval gate. **Regression test (named):** asserting `readFocusedFingerprintAndDomain` (and any pre-approval preflight read) returns no `value` field AND its evaluated in-page script result carries no candidate plaintext (e.g. the metadata script is asserted not to read selection/`.value`/`.innerText`, or the CDP-result object is asserted to have no `value` key), proving the preflight reads no value before approval. The captured candidate is then `Buffer.from(capture.value, "utf8")`'d and HMACed at the sink, with no value-bytes lingering across approval. (If a single-capture split is infeasible against the `BrowserOps` contract, document compare's pre-approval `capture.value` window as an additional **accepted long-lived string boundary** in §5 success-criterion (ii) alongside the child-env window — but the late-capture reorder is preferred and is the spec's requirement.)

**Sink reality (verified — not yet all Buffer-capable).** Only *some* sinks accept/scrub Buffers today: spawner child stdin, run-resolve dispose, and keychain `.fill(0)`. The core browser sinks are still **string-only** and must be classified, not assumed:

- **Buffer-convertible — internal APIs, 5q makes them Buffer-native (no wire-format change):**
  - **Child stdin** (`templates/run.ts`) — already buffers internally: `Buffer.from(input.secret, "utf8")` at `run.ts:110`, scrubbed in the `.end(buf, cb)` callback. Change the `TemplateRunInput.secret` contract to bytes (`SecretValue`/`Buffer`) so no string is materialized before the stdin write. (The in-code comment at `run.ts:100-103` already flags this as the "Phase 5q" refactor.)
  - **Tmp env-file writer** (`tmp-env-file.ts`) — `WriteSecretEnvFileInput.value: string` is converted to a Buffer immediately (`Buffer.from(\`${name}=${value}\n\`, "utf8")` at `tmp-env-file.ts:49`) and zeroed after write. Change the `value` field to bytes and build the `NAME=` prefix + newline as bytes so the value never exists as a JS string here.
  - **Masker** (`run/masker.ts`) — `createMasker(secrets: readonly string[])` immediately encodes each to a Buffer (`Buffer.from(s, "utf8")` at `masker.ts:58`) and already operates entirely on bytes (process/flush/dispose are byte-level). Change the constructor to accept bytes (`Buffer[]`/`SecretValue[]`) so the secret never enters the masker as a string.
  These are all daemon-internal function signatures, not HTTP/CDP wire formats — converting them to bytes is in scope for Tier A and removes avoidable string copies.
- **Unavoidable string boundary — child-process environment (`run-resolve.ts`), accepted + mitigated.** `run-resolve` resolves refs and assigns each into a child `env` object: `env[entry.key] = record.value` (`run-resolve.ts:337`), where `env: NodeJS.ProcessEnv` (`run-resolve.ts:327`) is passed to `spawnAndStream` (`SpawnInput.env: NodeJS.ProcessEnv`, `spawner.ts:21`). **`NodeJS.ProcessEnv` is string-valued by definition** (`Record<string, string | undefined>`) — the OS/`child_process` env API only accepts strings, so the resolved secret is **necessarily** materialized as a JS string here. This is a distinct accepted boundary (not browser/CDP, not vault read/write). 5q's discipline: hold the `ResolvedSecret`/`SecretValue`s, convert to a string **only at the `env[entry.key] = ...` assignment** (the latest possible point), build the masker from `.bytes()` (the masker is Buffer-native — see above), and `dispose()` each `SecretValue` once the child is spawned and the env strings are no longer needed. **Concrete drop-reference requirement (verified — a bare `dispose()`-after-`spawnAndStream` is insufficient).** Disposing the `SecretValue`s does NOT remove the secret *strings* assigned into the env object, and those strings outlive a naive spawn because **`spawnAndStream(input)` retains the whole `input` object — including `input.env` — in the Promise closure that drives the stdout/stderr/exit handlers** (`spawner.ts:65`, `spawner.ts:110-111`, `spawner.ts:175-191`); the closure lives for the child's entire lifetime, so `input.env` (and its secret string values) is reachable that whole time even after the daemon "no longer needs" it. 5q therefore requires two explicit drops, not one:
  1. **In `spawnAndStream`:** after the synchronous `spawn(input.cmd, input.args, { env: input.env, ... })` call returns (`spawner.ts:70`), the function must **not keep `input` (and thus `input.env`) reachable from the long-lived handlers**. Destructure the fields the handlers actually use (`outputWriter`, `signal`, `stdinBytes`) into locals up front and reference only those locals inside the `c.stdout`/`c.stderr`/`c.on(...)`/abort callbacks, so the `input` object — and its `env` — becomes unreachable as soon as `spawn` has copied the env into the child. (`child_process.spawn` reads `options.env` synchronously during the call, so dropping the daemon-side reference immediately after spawn does not affect the child.)
  2. **In the `run-resolve` route (the caller):** immediately after `spawnAndStream` is *initiated* (synchronously after the call returns / the spawn is kicked off — not after awaiting child exit), clear the secret entries out of the env object the route built (`delete env[entry.key]` for each resolved key, or null them) and `dispose()` each resolved `SecretValue`, so the route does not hold the secret env strings across the awaited child lifetime either. The masker bytes are scrubbed in the existing `finally`.

The remaining string copies are then only the ones the kernel handed to the child at spawn time, which live for the child's lifetime inside the *child* process — an inherent property of passing secrets via environment, the same limit that makes the masker necessary; the *daemon* heap no longer retains them past spawn. **Tests assert (a) each resolved `SecretValue` is disposed after `spawnAndStream` is invoked, and (b) `spawnAndStream` does not retain a reachable reference to the passed `env` object after spawn** (e.g. the route's env object can be cleared right after the call without affecting the child, or a spy confirms the handlers reference destructured locals, not `input`). Re-typing the env values to bytes is impossible (the platform API is string-only); the env-as-secret-channel is the user-chosen transport for `run`.
- **Unavoidable string/protocol boundary — browser/CDP + template-render-out (accepted, documented).**
  - **Browser/CDP:** `BrowserOps.injectFocused`, `proveAbsence`, and `injectIntoBackendNode` take `string`; `readBackendNodeValue`/`resolveWithinContainer`/`captureFocused`/`captureSelection` return `string` (`src/daemon/chrome/internal-ops.ts`); CDP injection/absence builds its in-page JS via `JSON.stringify(value)` (a string by construction). At this boundary the plaintext is **necessarily** re-materialized as a `string` for the CDP-script protocol, on both the inject (out) and capture (in) directions.
  - **Template render (`inject-render.ts`):** `ParsedTemplate.render(values: Map<string,string>): string` is **string-based by contract** (`template.ts:24`) — the parser substitutes refs into the template and returns a plaintext `string`. So `inject-render` must build a `Map<string,string>` at the render call. **Mode split:** in **stdout mode** (`-o -`) the rendered string is returned in the response `content` (`inject-render.ts:108`) — the existing, agent-requested plaintext-out wire surface noted in §0; in **file mode** (`-o <abs-path>`) the rendered string is written to a 0600 file and never returned. In both modes the consumer holds a `ResolvedSecret`, converts each `SecretValue` to a string **only at the `valuesMap.set(...)` call**, renders, then disposes each `SecretValue`; the rendered string and the per-value strings are the bounded transient. **Required file-mode reorder (verified — current order holds the render string across async filesystem work).** Today `inject-render` builds `valuesMap` + `render`s at `inject-render.ts:93-97` **before** the file-mode path-safety walk, so in file mode the `rendered` string is held across the entire async `realpath`/`lstat`/`mkdir`/`open` sequence (`inject-render.ts:137`, `:174`, `:192`, `:203`, `:223`, `:239`) before the `writeFile` at `inject-render.ts:240` — i.e. for human-/IO-latency-long, contradicting "materialized only at the sink call and dropped right after." 5q therefore requires **moving all file-mode path validation/setup (the absolute-path check, the ancestor-symlink walk, `realpath` HOME-containment checks, step-wise `mkdir`, the leaf-symlink check, the final TOCTOU re-`realpath`, and the temp-file `open`) to run BEFORE the `valuesMap` construction + `render`**, so that in file mode the secret values are turned into the `Map<string,string>` and the template is rendered **immediately before `writeFile`** — with the rendered string and per-value strings disposed/dropped right after the write resolves. (Stdout mode is unaffected — it returns immediately after render; only the file-mode branch reorders so render is the last step before the sink.) Note the stdout-mode early-return at `inject-render.ts:99-109` must stay reachable: the reorder keeps the `outputPath === "-"` branch rendering-then-returning, and moves only the *file-mode* setup ahead of the file-mode render. Re-typing `render`/the template contract to bytes would change a daemon-internal-but-string-by-design contract — deferred with the browser/CDP items (§3).

  Re-typing the browser/CDP APIs would change the `BrowserOps`/CDP read/write contract; re-typing `render` would change the template contract — both deferred with Tier B (§3).

5q's discipline at the browser/CDP boundary: hold the `SecretValue` as long as possible, convert to a string **only at the call to the sink** (`sv.bytes().toString("utf8")`), and dispose the `SecretValue` immediately after the awaited sink resolves. The transient string there is short-lived and unavoidable without changing the `BrowserOps`/CDP contract (out of scope here — see §3, alongside Tier B). **The §2 audit (Task 1) classifies every consumer as Buffer-native vs. accepted-string-boundary so the implementer does not silently recreate a long-lived plaintext string at a sink the spec marked Buffer-capable.** Where a sink is a daemon-internal API (stdin, env-file, masker), 5q makes it Buffer-native; the accepted-string-boundary sinks that keep a transient are the **browser/CDP inject+capture boundary**, the **`inject-render` template render** (`Map<string,string>` + rendered string — file-mode rendered immediately before `writeFile` per the reorder above), and the **`run-resolve` child `NodeJS.ProcessEnv`** (string-only platform API; dropped from the daemon heap right after spawn per the drop-reference requirement above) — the documented limits of Tier A. Of these, the browser/CDP and template-render transients are dropped right after the sink call; the child-env strings are dropped from the daemon at spawn but inherently survive inside the child for its lifetime.

### Fingerprint signature

```ts
export function fingerprintSecret(value: Buffer, key: Buffer): string;       // was (value: string, key)
export function fingerprintMatches(value: Buffer, fingerprint: string, key: Buffer): boolean;
```
Internally HMACs the bytes (identical output to the old string form for the same bytes — **no stored-fingerprint migration**). Callers pass bytes per their source:
- **Vault-internal callers** that currently pass `stored.value` (a string, e.g. fingerprint-recompute on the stored record) pass `Buffer.from(stored.value, "utf8")`.
- **The compare route does NOT pass a resolved/stored secret.** It reads only the stored record's `fingerprint` field (metadata) and HMACs the **browser-captured candidate** — `fingerprintMatches(capture.value, secret.fingerprint, fpKey)` (`secrets.ts:515`), where `capture.value` is the page-side string. Migrate that one argument to `Buffer.from(capture.value, "utf8")` (the capture string is the same accepted CDP/protocol boundary as the other browser sinks — see §2 capture-shape note). **Critical: compare must keep HMACing the captured candidate, never a stored/resolved secret — HMACing the stored secret against its own fingerprint would make `matches` trivially true and stop comparing against the page value.**
- **True resolved-plaintext consumers** (inject etc.) pass `resolved.value.bytes()`.

### Inbound (write) path

`UpsertSecretInput.value` becomes `SecretValue` (the value being stored). Producers — **all `upsertSecret` call sites (verified — the §2 audit must re-confirm the final map):**
- **Generated-value producer shape (verified — `generateSecretValue` returns an *encoded string*, not raw bytes).** `generateSecretValue(kind)` (`src/daemon/helpers/generate-value.ts:5`) returns `randomBytes(32|64).toString("base64url" | "hex")` — i.e. an **already-encoded JS `string`** (base64url or hex), which is the value's canonical stored shape. So the two `Vault.generate`/generate-route producers below are **`SecretValue.fromUtf8(generateSecretValue(kind))`** producers (wrap the encoded string), **NOT** `SecretValue.fromBuffer(randomBytes(...))` — passing the raw 32/64 random bytes via `fromBuffer` would change the stored value (and its fingerprint) from the base64url/hex shape callers expect to raw binary, a behavior change and a serialization hazard (the on-disk `.value` is `.bytes().toString("utf8")`). The bytes inside this `SecretValue` are the **encoded characters**; `SecretValue.fromUtf8` preserves the exact output `generateSecretValue` produces today. (Audit-pass classification: accepted-encoded-string producer, wrap-and-drop — the encoded string is consumed immediately into `fromUtf8` and the local dropped.) (JS string immutability means these originals can't be zeroed; the win is making them unreachable promptly, not erasing them — same Tier-A bound as the vault read transient.) (The `entries[].key` field stays a plain string — it is not a secret.)

This keeps reveal-capture's plaintext lifetime entirely inside the already-enumerated boundaries (the browser/CDP `proveAbsence` transient + the vault read/write op) with **no new long-lived string window to enumerate**, and avoids the use-after-dispose hazard. (The alternative — keeping `capturedValue` as a page-side string across the upsert and wrapping a *separate* `SecretValue.fromUtf8(capturedValue)` for the store — is acceptable only if the proof-before-upsert reorder is infeasible against the route's audit/blind-resume ordering; if taken, the page-side `capturedValue` string window must be named in §5 success-criterion (ii) as an additional accepted browser/CDP-class string boundary. The reorder is the spec's requirement precisely to avoid that.)

**`Vault.generate` return type — change to no-value metadata (required by the type split).** Today `Vault.generate(...)` returns `Promise<SecretRecord>` by reading the record back via `return this.getSecret(ref)` (`vault.ts:206`), so it currently hands a **string-valued `SecretRecord` across the vault boundary** to its caller (`secrets-rotate.ts:83`) — which contradicts the §2 invariant that the stored string `SecretRecord` is vault-internal only. **Browser capture shape (verified — currently `string`, not bytes).** Capture does NOT return bytes today: `CaptureResult.value` is typed `string` and `BrowserOps.captureFocused`/`captureSelection`/`readBackendNodeValue`/`resolveWithinContainer` all return `string` (`src/daemon/chrome/internal-ops.ts`), because the value crosses the CDP boundary as a JSON string. So `reveal-capture` and bootstrap-capture **wrap an existing string** (`SecretValue.fromUtf8(captured.value)`), not raw browser bytes — the capture-side string is an **accepted protocol boundary** (same class as the inject string boundary in the consumer section), NOT a Buffer-native path. The audit must record each capture site as an accepted-string-boundary producer; making the capture APIs Buffer/`SecretValue`-native would require changing the `BrowserOps`/CDP read contract and is deferred with the other string-boundary items.

**`upsertSecret` owns and disposes `UpsertSecretInput.value` on every exit path (the hard guarantee).** `upsertSecret` serializes `input.value.bytes().toString("utf8")` into the stored `SecretRecord.value` string (Tier A — on-disk stays string). But `upsertSecret` can throw **before** it ever writes — most notably the duplicate-ref `secret_exists` check (`vault.ts:66`) throws before `record` is built, and `this.read()`/`this.write()` can throw too — so a happy-path-only `dispose()` would leak the inbound `SecretValue`'s bytes on those throws. The spec therefore makes the **vault boundary** the disposal guarantee: `upsertSecret` **takes ownership of `input.value` and disposes it in a `finally`** so the inbound `SecretValue` is scrubbed on the successful-write path, the `secret_exists` throw, and any read/write error alike. (Defensive: because `dispose()` is idempotent and `.bytes()`-after-dispose throws, route-level producers may *also* dispose their `SecretValue` on their own error/finally paths — e.g. import's unconsumed-entry disposal — without double-free hazard; but the vault `finally` is the authoritative scrub, so a producer that hands its value to `upsertSecret` need not separately dispose that handed-off value.) The transient `toString("utf8")` written into the stored string is the same bounded encrypt-boundary transient described above.

  - **run-resolve clears env entries after spawn init:** assert that the route nulls/deletes the secret env entries (`delete env[entry.key]`) and disposes each resolved `SecretValue` immediately after `spawnAndStream` is initiated (not after child exit), and that `spawnAndStream` does not retain a reachable reference to the passed `env` object after `spawn` (e.g. clearing the route's env right after the call does not affect the running child, or a spy confirms the long-lived handlers reference destructured locals rather than `input`).
  - **inject/inject-submit/templates-stdin/run-resolve/inject-render resolve plaintext only after approval:** for each of these consumers, with a stubbed approval that records call order (or a deferred approval), assert the plaintext-resolving accessor (`resolveSecret`/`SecretValue`-returning `resolveRefs`) is invoked **after** `requireApprovals` resolves — and that the pre-approval policy/domain/binding preflight used only the no-value metadata accessor — so no `SecretValue` (live plaintext bytes) exists in the daemon across approval latency; and assert each `SecretValue` is disposed in the route's `finally` on the success, throw, and timeout paths. (`inject-render` must be included explicitly: today it resolves refs **before** the conditional production approval gate — `resolveRefs` at `inject-render.ts:54`, `requireApprovals` at `inject-render.ts:65`/`:80` — so its late-resolve ordering regresses on the same conditional-gate boundary as `run-resolve`; "after approval" = after the `isProduction` block whether or not it awaited.)
  - **inject-submit retains one `SecretValue` across both sinks and disposes once in the outer `finally`:** assert that the **same** `SecretValue` instance feeds both `injectIntoBackendNode` (`inject-submit.ts:171`) and, after the up-to-`successTimeoutMs` `observeText` (`inject-submit.ts:215`), `proveAbsence` (`inject-submit.ts:222`) — i.e. it is **not** disposed after the inject — and is `dispose()`d exactly once in the route's **outer** `finally` (assert `.bytes()` succeeds at the `proveAbsence` sink but throws after the route returns, on the success, throw, and `successTimeoutMs`-timeout paths alike). This pins the deliberate multi-sink retention window.
  - **reveal-capture proves absence before upsert with no use-after-dispose:** assert the proof-before-upsert ordering — `proveAbsence` (`reveal-capture.ts:436`) is invoked **before** `upsertSecret` (`reveal-capture.ts:424`) on the single owned `SecretValue`, the same `SecretValue` is then handed to `upsertSecret` (which disposes it in its `finally`), and no `.bytes()` call throws "used after dispose()" on the happy path (capture + proof + store all succeed). (If the separate-upsert-owned-copy alternative is taken instead, assert the store-copy `SecretValue` is disposed by `upsertSecret` and the page-side string is dropped right after the proof resolves.)
  - **`upsertSecret` disposes the inbound `SecretValue` on every exit path:** assert the passed `UpsertSecretInput.value` is disposed (`.bytes()` throws / `dispose` spy fired) after a **successful** upsert AND after a **throwing** upsert — specifically the duplicate-ref `secret_exists` throw (`vault.ts:66`, force omitted) — so a pre-write throw still scrubs the handed-off value.

## §5 — Success criteria

1. **5s:** `secret-shuttle init --per-project-identity` in two different project dirs on the same machine produces two different agent ids; without the flag, the id is byte-identical to today's. Existing users (no opt-in) see zero identity change.
2. **5q:** no secret value is held as a long-lived JS `string` on the per-secret use path *beyond the accepted platform/protocol boundaries below* — consumers operate on `SecretValue.bytes()` and dispose after use; metadata/existence callers hold no plaintext at all (no-value accessor); no daemon route retains a *stored* plaintext string after the resolve boundary. The only surviving plaintext strings are the bounded ones explicitly enumerated + classified in §2: (i) the vault read/write op (Tier B); (ii) the **browser/CDP** inject+capture transients and the **template render** transient (`inject-render`'s `Map<string,string>`/rendered string), each materialized only at the sink call and dropped right after; (iii) the **child env** values (`run-resolve` → string-valued `NodeJS.ProcessEnv`), which by the platform env API necessarily live for the child's lifetime; and (iv) the **agent-requested `inject-render -o -` render**, which intentionally leaves the daemon in the response `content` (the one existing plaintext-out wire surface, §0). Each `SecretValue` is disposed once its sink no longer needs it (env: after spawn; render/inject/capture: after the awaited sink resolves). **No candidate/secret value is held across an approval gate (every route, not just compare/import):** each true-plaintext consumer runs only the no-value metadata preflight before `requireApprovals` and resolves the plaintext `SecretValue` only afterward, immediately before its sink (the two-phase late-resolve discipline in §2 "Consumer changes") — `inject-submit`, `/v1/secrets/inject`, `templates` (stdin), `run-resolve` (env), and `inject-render` no longer hold a resolved `SecretValue` across approval; `compare` captures the page candidate only *after* `requireApprovals` resolves (only the no-value domain/fingerprint preflight precedes approval); and `import` wraps each value into a `SecretValue` and drops the originally-parsed request-body strings at the parse loop *before* its production approval gate — so no candidate or resolved secret value lives across human-approval latency on any route. Accidental stringification (`${}`, `JSON.stringify`, log) of a `SecretValue` yields `"[secret]"`, proven by test. Existing stored fingerprints remain valid (no migration). The on-disk vault format is unchanged.
3. **No wire-format change:** every HTTP request/response shape is identical to v0.3.1 (5q is internal-only).
4. **5a doc accuracy:** keychain `index.ts`/`types.ts` comments describe the shipped `@napi-rs/keyring` reality, not "stubs."
5. **Full suite green** (1588 baseline + new tests, 0 fail), `tsc --noEmit` clean, `check-pack` OK. v0.4.0 in `package.json`.
6. **All three codex gates clean** (spec, plan, impl).

## §6 — Risks

| Risk | Mitigation |
|---|---|
| A missed `.value` consumer keeps passing a `string` into a sink that lingers — partial hygiene | The §2 audit pass (step 2) produces a definitive site map; the guard test scans for raw resolved-`.value` usage; code-quality review checks each consumer scrubs. |
| `SecretValue.toJSON → "[secret]"` accidentally corrupts a path that SHOULD serialize the real value (e.g. the on-disk write) | The type split keeps on-disk serialization on the *stored string* (`SecretRecord.value: string`), never on a `SecretValue`. The write path explicitly calls `.bytes().toString("utf8")` to produce the stored string. Test: a stored vault round-trips its real value on disk. |
| `dispose()` called too early (value scrubbed before an async sink finishes reading it) | Follow the existing spawner pattern (scrub in the write-callback / after the awaited sink resolves, not before). Per-consumer e2e tests assert the operation succeeds end-to-end. |
| 5s opt-in flag clobbers an existing `secret-shuttle.config.json` `infer.*` block | The flag MERGES `identity.perProject` into the existing config (preserving other keys); test covers a config that already has `infer.supabaseNames`. |
| `git rev-parse` behaves unexpectedly (submodule, worktree, detached) → wrong scope | `resolveProjectScope` only needs a stable absolute path; `--show-toplevel` returns the worktree root, which is stable per checkout. Non-repo / error → cwd fallback. Documented; finer cases are the deferred sub-monorepo item. |
| Fingerprint signature change breaks a caller that still passes a string | TypeScript catches every call site at compile time (string → Buffer is a type error); the migration-free pin test guarantees identical output for the same bytes. |
